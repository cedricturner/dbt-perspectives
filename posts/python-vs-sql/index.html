Architecture

Python or SQL
is the wrong question

The language doesn't matter. Where the computation runs does. When you transform data in Python, you're solving a different architectural problem than language preference — and creating a different set of constraints.

What people think

The debate resurfaces regularly. Python is more expressive — you can use libraries, write complex logic, do things SQL simply can't. SQL is more readable, closer to the data, easier to audit. Teams pick sides, tooling follows, and the real question gets missed.

The real question isn't about syntax or expressiveness. It's architectural: where does the computation actually run?

Language is a preference. Location is a constraint. And most teams optimizing for language preference end up surprised by the constraints they've inherited.

The tension

When you write a Python transformation, the computation runs on a machine — yours, or a container. The data travels from the warehouse to that machine, gets processed in memory, and the result travels back. Your machine's resources set the ceiling.

When you write a SQL transformation in dbt, the computation runs inside the warehouse. The data doesn't move. The warehouse's compute and memory handle it. You can resize those resources without changing the code.

The difference isn't style. It's where the work happens — and what that implies about scale, failure, and observability.

Python — compute leaves the warehouse
dbt — compute stays in the warehouse

Same transformation. Different architecture. One pulls data to a machine; the other keeps it in the warehouse.

The language you write in is secondary. The fundamental question is: where does the computation happen? On your machine, where RAM is the ceiling? Or in the warehouse, where scale is a configuration?

What actually changes

The architectural choice determines a cluster of downstream properties — not just scale, but also persistence, parallelism, observability, and cost model. These aren't independent variables. They follow from the location decision.

compute on machine (Python)

⚠
Scale: ceiling is machine RAM. 10GB often requires rewrite.
⚠
State: DataFrames vanish on crash. Debug requires re-run.
⚠
Parallelism: manual. Threading adds complexity.
⚠
Cost: pay for machine uptime + warehouse storage separately.
✓
Flexibility: arbitrary libraries, ML models, complex iteration.
✓
Control: full execution environment, no warehouse limits.

compute in warehouse (dbt/SQL)

✓
Scale: resize the warehouse. Same SQL at 100MB or 10TB.
✓
State: intermediate tables persist. Debug without re-run.
✓
Parallelism: automatic from ref() dependency graph.
✓
Cost: pay for warehouse compute only. No separate infra.
⚠
Flexibility: set operations only. ML, complex iteration → Python.
⚠
Control: warehouse determines query planner behavior.

What emerges

Once you frame the decision as "where should this computation run?" instead of "which language should I use?" — the answer is usually obvious. Most data transformation is set operations on structured tables. Warehouses are purpose-built for that. Run it there.

The places where Python genuinely makes sense are narrower than people assume — and narrower than the Python ecosystem's popularity suggests.

task where it should run why
Filter, join, aggregate structured tables warehouse / dbt Warehouse is purpose-built for this. Scales without code changes.
Train or run an ML model Python Requires libraries (sklearn, torch) that don't exist in SQL.
Call an external API to enrich rows Python Network I/O doesn't belong in the warehouse.
Parse and normalize free-text fields depends on complexity Simple regex → SQL. Complex NLP → Python.
Generate a feature table for model training warehouse / dbt Feature engineering is SQL. Training is Python. Do both.
Reshape / unpivot data at scale warehouse / dbt Modern warehouses handle PIVOT/UNPIVOT. No RAM limit.
Complex iteration / custom aggregation logic Python SQL's set model doesn't handle row-by-row iteration well.
Write cleaned data from an API to a warehouse Python Ingestion is Python (or an EL tool). Transformation is dbt.

The pattern: ingestion and operations that require external compute → Python. Anything that runs cleanly as a set operation → warehouse.

When it matters

The location decision is abstract until scale forces it. Click each scenario to see where the tradeoff becomes concrete.

Data volume grows 10x over a year▼
Python pipeline

The script starts hitting memory limits. The fix is chunking, Dask, or rewriting to push the logic into the warehouse anyway — which means rewriting the transformation in SQL. You did the work twice.

dbt

Same SQL. Resize the warehouse if needed. The transformation is already in the warehouse — there's no architectural ceiling to hit. The cost model changes; the code doesn't.

The hidden cost: Python pipelines often get migrated to SQL when data grows. If you anticipate that, starting in SQL avoids a rewrite.
You need to run ML on top of the transformed data▼
all-Python approach

The feature engineering and model training are mixed in the same script. It's hard to test the features independently. When the model is retrained, the feature logic re-runs too — even if nothing changed.

dbt + Python

Feature engineering in dbt (warehouse). Training in Python using ref('fct_features') as input. The feature table is a persistent, testable warehouse table. Python only handles what SQL can't.

dbt and Python aren't alternatives — they're layers. dbt produces the features; Python consumes them. Both do what they're good at.
A new person joins who doesn't know Python▼
Python pipeline

They need to understand the Pandas idioms, the DataFrame merges, the order of operations. They can't query intermediate results without re-running the script. The logic is implicit in the code structure.

dbt

Each model is a SQL SELECT. Any analyst with SQL skills can read it, query the intermediate tables, and understand what it does without running anything. The pipeline is self-documenting by structure.

SQL is more widely readable than Python. For data pipelines that more than one person needs to understand, that matters.
A pipeline fails at 2am▼
Python pipeline

Process crashed. No intermediate state survived. You need to reproduce the failure, which means re-running the whole script — same data transfers, same compute time — before you can diagnose. You're debugging from a blank slate.

dbt

The completed models are still in the warehouse. The warehouse has a query log showing exactly what ran. You re-run only the failed model. Debugging starts from evidence, not from scratch.

At 2am, the question isn't "which language is better." It's "what can I inspect right now, without re-running anything?"

The point

Python or SQL is the wrong question.

The question is: where should this computation run? On a machine, with your machine's memory as the ceiling, intermediate state that vanishes on failure, and data that has to travel back and forth? Or in the warehouse, where the data already lives, at the scale the warehouse provides, with intermediate results that survive as queryable tables?

Python and SQL aren't alternatives — they're tools for different layers. Python for ingestion, ML, and operations that require external compute. SQL (in dbt) for transformation that can be expressed as set operations on data that's already in the warehouse.

The teams that argue most intensely about Python vs SQL are usually the ones who haven't yet hit the scale at which the location decision becomes obvious. Once you do, the language preference feels secondary.

The language debate is a proxy for a location debate.
Once you see it that way, the answer is usually obvious: run computation where the data lives, use Python for what SQL genuinely can't do.
That's not a Python vs SQL opinion. It's a principle about where work should happen.
