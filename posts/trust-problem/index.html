Adoption & Culture

Data teams don't have
a quality problem.
They have a trust problem.

Tests tell you when something broke. They don't make the pipeline legible. Most teams rebuild the same model five times — not because the first one was broken, but because they couldn't tell whether it was right.

What people think

The standard response to "I don't trust the data" is: add more tests. Write assertions. Check row counts. Test uniqueness. Expand coverage. It's an engineering instinct — if something can break, add a guard.

It's not a wrong instinct. But it's aimed at a different problem than the one most teams actually have.

Most data teams don't distrust their pipelines because they produce wrong data. They distrust them because they can't tell whether the data is right. Those are different problems — and they have different solutions.

The tension

Here's the canonical symptom: two teams are presenting to the same executive. Revenue for Q3. Different numbers. Both teams built their models carefully. Both are passing their tests.

The meeting turns into a definition negotiation. Whose number is right? What counts as revenue — gross, net, recognized? Are refunds subtracted? How does it handle the month-end edge case? Nobody knows, because the pipelines are opaque — you'd have to read the scripts to find out.

team Q3 revenue tests definition edge case

Five teams. Five numbers. All passing tests. Nobody wrong — they just answered different questions.

Tests confirmed each model was internally consistent. They didn't — and couldn't — tell anyone which model was canonical. That's a visibility problem, not a quality problem.

What actually changes

Tests and visibility are different capabilities. Tests verify correctness: when a constraint is violated, you find out. Visibility makes the pipeline legible: anyone can inspect what it does, trace intermediate steps, and understand the definition without reading the source code.

what tests give you

✓
Correctness signals — know when a constraint breaks
✓
Regression detection — know when an update breaks something
✗
Can't tell you if the logic is right, only if it's internally consistent
✗
Don't tell another team what the model actually does
✗
Don't prevent two teams from defining "revenue" differently

what visibility gives you

✓
Inspectability — anyone can query intermediate steps
✓
Reusability — others can build on your model instead of rebuilding it
✓
Single definition — ref('stg_orders') everywhere means one answer
✓
Transfer of trust — "I trust it because I can see it" extends beyond the original author

The shift dbt enables isn't primarily the test framework. It's that every intermediate step is a real, persistent, queryable table. When another team wants to understand your revenue model, they don't schedule a meeting — they run a SELECT.

What emerges

When the canonical definition lives in one dbt model — and every team references it with ref() — there's no longer a "which revenue?" meeting. There's one answer. It might still be wrong in some people's eyes, but at least the argument is about the definition, not about whose calculation to trust.

models/marts/fct_revenue.sql — one canonical model all teams reference this
-- The definition lives here. Change it once, every ref() gets the update. SELECT o.order_id, o.close_date, o.total_amount - COALESCE(r.refund_amount, 0) AS net_revenue, o.is_recognized FROM {{ ref('stg_orders') }} o LEFT JOIN {{ ref('stg_refunds') }} r ON o.order_id = r.order_id WHERE o.status = 'closed' AND o.is_recognized = true
Every team that needs revenue references this model:
ref('fct_revenue') — Finance ref('fct_revenue') — Growth ref('fct_revenue') — Sales Ops ref('fct_revenue') — Analytics ref('fct_revenue') — Data Eng

Five teams. One number. The definition debate becomes a PR review — not a recurring meeting.

The "five versions of revenue" problem isn't a definition problem. It's a visibility problem. When there's one canonical table that everyone can query and inspect, the question shifts from "whose number?" to "is this definition right?" — which is a much more productive conversation.

The consequence

The downstream effect of opaque pipelines isn't usually a data quality failure. It's a duplication failure. Teams stop trusting shared infrastructure and start maintaining their own private copies.

Over time, the organization ends up with five "revenue" definitions, three "customer" definitions, and a permanent low-grade argument about whose numbers are right. Every new analyst who joins inherits this confusion. Every new dashboard adds to it.

The fix isn't more tests. Tests don't prevent duplication — they just make each duplicate slightly more reliable. Visibility prevents duplication, because when people can inspect what a model does, they build on top of it instead of alongside it.

When it breaks

Trust failures look different from quality failures. Click each scenario.

A number looks off in a dashboard▼
opaque pipeline

Alert fires. The on-call analyst checks the dashboard and can't tell where the number comes from. They Slack the data engineer who built it. Three hours later: "looks like a data quality issue." Nobody can see inside the model to confirm.

dbt — visible pipeline

Query fct_revenue. Does the row count look right? Query stg_orders for the same date range. The discrepancy is visible without a human expert to mediate. The model is the documentation.

The difference: in an opaque pipeline, debugging requires the person who built it. In a visible pipeline, anyone who can write SQL can investigate.
New analyst joins and needs to understand the data▼
opaque pipeline

The onboarding doc says "use the revenue table." The analyst queries it and gets a number that doesn't match last week's exec review. They schedule a meeting with three people to understand why. This happens four times in their first month.

dbt — visible pipeline

The analyst reads fct_revenue.sql. They can see exactly what's included and excluded. They can trace upstream to stg_orders and see where the data comes from. Questions answered without a meeting.

Every meeting to "align on definitions" is a visibility failure that should have been a SQL file.
Two teams have different numbers in the same exec review▼
opaque pipeline

The meeting stalls. Someone says "let's take this offline." An action item is created to "align on revenue definition." This action item recurs quarterly. Nobody has time to fix it. The distrust compounds.

dbt — visible pipeline

Both teams reference ref('fct_revenue'). They get the same number. If there's disagreement about whether that number is right, it's a discussion about the model definition — which has a PR, a history, and a clear owner.

The difference: one shared model makes disagreements concrete ("the model excludes refunds — should it?") instead of diffuse ("we have different numbers — why?").
A staging model gets updated▼
opaque pipeline

The analyst updates their script. Two weeks later, three dashboards start showing different numbers. Nobody knows what changed or which reports are affected. A post-mortem is scheduled to "trace the lineage manually."

dbt — visible pipeline

dbt's lineage graph shows exactly which downstream models reference stg_orders. Run dbt run -s stg_orders+ to re-execute everything that depends on it. Impact is visible before and after the change.

Lineage is a byproduct of visibility. You don't build it — it emerges from ref() calls that already had to exist.

The point

Data teams don't have a quality problem. They have a trust problem.

And trust isn't built by adding more tests to the same opaque pipeline. Trust is built when people can look inside the pipeline — when the intermediate steps are real tables they can query, when the definition lives in one file they can read, when a new analyst can understand the model without booking a meeting.

Tests are necessary. They're not sufficient. A pipeline can pass all its tests and still be trusted by nobody — because trust requires legibility, and legibility requires visibility.

The question to ask about any data platform: can a person I've never met understand what this model does by reading it? If yes, trust can transfer. If no, it's locked in the head of whoever built it.

Most data teams don't rebuild because the platform is unreliable. They rebuild because they can't see inside it.
"I trust it" usually means "I built it." That's not a quality problem — it's a visibility problem.
Making intermediate state persistent and visible doesn't improve data quality. It transfers trust.
