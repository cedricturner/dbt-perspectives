Data Engineering

What moving to dbt
actually changes

It's not really about the SQL. And it's not about version control or modular files. What changes is where the computation actually runs. That one thing — your machine vs the warehouse — changes pretty much everything else.

What people think

The comparison gets framed as a language debate. You have a Python pipeline, someone says dbt is just SQL, and suddenly the evaluation becomes: what does your team prefer? What does your data require?

Teams with solid Python pipelines often dismiss dbt early. "We already have modular functions. We already have unit tests. Why rewrite everything?"

It's a reasonable question. It's aimed at the wrong thing.

The tension

The language isn't what matters here. The location is.

A Pandas pipeline pulls data to your machine, transforms it in memory, and writes the result back. Every pd.read_sql() is a data transfer. Every DataFrame exists only while the process runs. When it fails, everything is gone.

build_customer_order_summary.py — pandas pipeline 1 file · 6 steps 
import pandas as pd
from sqlalchemy import create_engine

engine = create_engine("postgresql://...")

# Step 1 — pull ALL rows to this machine  ⚠ data transfer — grows with table size
raw_customers = pd.read_sql("SELECT * FROM raw.customers", engine)  ⚠ hardcoded schema
raw_orders    = pd.read_sql("SELECT * FROM raw.orders",    engine)  ⚠ hardcoded schema
raw_payments  = pd.read_sql("SELECT * FROM raw.payments",  engine)  ⚠ hardcoded schema

# Step 2 — transform in memory  ⚡ DataFrame — gone if process crashes
clean_customers = raw_customers.copy()
clean_customers["email"]      = clean_customers["email"].str.strip().str.lower()
clean_customers["first_name"] = clean_customers["first_name"].fillna("Unknown")
clean_customers["last_name"]  = clean_customers["last_name"].fillna("Unknown")

# Step 3 — filter orders (in memory)  ⚡ DataFrame — not queryable after crash
clean_orders = (raw_orders
    .loc[raw_orders["status"] != "cancelled",
         ["order_id", "customer_id", "order_date", "total_amount"]])

# Step 4 — filter payments (in memory)  ⚡ DataFrame — serial, must wait for step 3
clean_payments = (raw_payments
    .loc[raw_payments["amount"] > 0]
    .rename(columns={"amount": "payment_amount"})
    [["payment_id", "order_id", "payment_amount"]])

# Step 5 — join
pmt_by_order     = clean_payments.groupby("order_id", as_index=False)["payment_amount"].sum()
orders_w_pmts    = clean_orders.merge(pmt_by_order, on="order_id", how="left")

# Step 6 — aggregate and write back to warehouse
summary = (clean_customers  ⚠ full replace every run
    .merge(orders_w_pmts, on="customer_id", how="left")
    .groupby(["customer_id", "email"], as_index=False)
    .agg(lifetime_orders=("order_id", "count"),
         lifetime_revenue=("total_amount", "sum")))
summary.to_sql("fct_customer_order_summary", engine,  ⚠ data travels back to warehouse
              schema="analytics", if_exists="replace", index=False)

One file. Data travels to your machine, gets transformed in memory, travels back. Nothing survives if the process dies.

This isn't bad code. It does exactly what it was built to do. The problem is the architecture. Computation happens on your machine, which means the data has to travel there — and your machine's memory is the ceiling.

What actually changes

dbt doesn't swap Python for SQL. It moves the computation from your machine into the warehouse — where the data already lives. The logic is nearly identical. What changes is where it runs.

Instead of pd.read_sql("SELECT * FROM raw.customers"), you write source('raw','customers') and the warehouse handles it. Instead of DataFrames in memory, each step becomes a real table. Instead of a script that disappears when it runs, models that leave evidence behind.

The pattern: pd.read_sql() → source(). DataFrame → ref(). to_sql() → gone. Data stops traveling. Computation moves to where the data already is.

What emerges

The Python script had this same structure inside it. The same dependencies existed — they were just hidden in the order of the code. Every ref() call makes one of those dependencies explicit. dbt reads them all and builds the execution order automatically.

All of this existed inside the Python script. It just wasn't visible — and it definitely wasn't parallel.

The consequence

The difference compounds at scale. A Python pipeline's ceiling is your machine's memory. A dbt pipeline's ceiling is your warehouse — which you can resize without changing any code.

❌ python pipeline — compute on machine

100 MB datasetworks fine
10 GB datasetslow, chunking needed
1 TB datasetOOM — rewrite required

debug intermediatere-run + print()

✅ dbt — compute in warehouse

100 MB datasetworks fine
10 GB datasetsame code
1 TB datasetresize warehouse, same code
intermediate statepersists as tables
parallel stepsautomatic from ref()
debug intermediateSELECT * FROM stg_orders

The code change between 100MB and 1TB: zero. The infrastructure change: warehouse size. That's the tradeoff.

When it breaks

This is where it actually matters. Not when things go well. When they break.

Pipeline OOMs on a larger dataset▼
python pipeline

MemoryError on the merge(). The DataFrame that caused it is gone. You re-run with a smaller sample, or you rewrite with chunking. Either way — analytics work stops while you fix a plumbing problem.

dbt

The query runs in the warehouse. There's no RAM ceiling. If the warehouse runs out of resources, you resize it. The code doesn't change.

The difference: one architecture hits a hard ceiling at your machine's RAM. The other scales with infrastructure — without touching code.
Wrong row count in the final output▼
python pipeline

Add print(len(df)) after each step, re-run the whole script, check the counts. Multiple cycles before you find where the row count changed. The intermediate DataFrames are gone — you have to recreate the state every time.

dbt

Query stg_orders. How many rows? Query int_orders_with_payments. How many? The difference tells you exactly where it happened. No re-run needed.

You can bisect a data problem the same way you'd bisect a software bug — because the intermediate state is still there.
Pipeline crashes mid-run▼
python pipeline

All DataFrames gone. Everything from steps 1–4 died with the process. You re-run from the top — same data transfers, same compute time — just to get back to where it was.

dbt

Every completed model is still in the warehouse. Run dbt run -s failed_model+ and only the thing that failed re-executes. Everything upstream is already there.

The difference: in-memory state disappears when the process dies. Warehouse tables don't. That determines whether you restart from zero or from where you left off.
Another team wants to reuse your clean orders logic▼
python pipeline

They copy your filter logic into their own script. Now there are two definitions of "valid order." When you update yours, theirs doesn't change. Nobody notices until the numbers diverge in a meeting.

dbt

They write ref('stg_orders'). They're building on your model directly — same table, same definition. When you update it, they get the update automatically.

The real value of ref() isn't dependency tracking. It's that the table gets reused instead of copied and redefined.

The point

Python or SQL is the wrong question.

The real question is: where does your computation run? On your machine — with your machine's memory as the ceiling, intermediate state that vanishes when the process dies — or in the warehouse, where the data already lives and the scale is handled for you?

For genuinely complex transformations — ML models, arbitrary iteration, things that actually need Python libraries — the answer might be "on a machine." Use Python for those.

For everything that can be expressed as a set operation — which is most of what data transformation actually is — the answer is almost always "in the warehouse." Not because SQL is better. Because running compute where the data already lives is just a simpler architecture.

The migration from a Python pipeline to dbt isn't a rewrite. It's a relocation.
The logic moves from your machine to the warehouse. Everything else — scale, parallelism, intermediate persistence — follows from that one change.
That's what the migration is actually about. The Python vs SQL debate is almost beside the point.
